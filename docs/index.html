Parallel agents collide.

Spawning many agents is easy. Making them cooperate is not. The moment two of them touch the same file, record, or decision, you inherit every hard problem in distributed systems, usually discovered as silent corruption or a mysteriously stuck run.

failure mode · silent

Clobbering

Two agents write the same state. Last write wins; the other's work vanishes with no error. You find out in review, or in production.

failure mode · expensive

Wasted work

An agent does expensive work, then discovers at commit that another already touched it. Redo. Burned tokens. Burned wall-clock.

failure mode · recurring

Amnesia

Knowledge learned in one run dies with it. The next run rediscovers the same facts from scratch, every time.

Four kinds of shared memory.

Agents join a Run and share four things through one MCP server. The vocabulary is deliberately small and precise.

Claim

A pre-flight declaration of intent over a Surface (a path like repo/src/auth or an entity like user:42). Shared or exclusive, leased, atomic. This is how an agent says “I’m about to touch this” before it does.

Blackboard

Live shared working state, keyed by Surface. Writes are enforced. You can’t write a cell without holding an exclusive claim on it. “Should not collide” becomes “cannot collide.”

Journal

An append-only, sequence-ordered history of everything that happened in a Run: every claim, write, and agent note. Fully replayable.

Fact

Durable, deduplicated, semantically searchable knowledge that outlives the Run. Learned once, reused forever, across sessions.

One process. One authoritative writer.

Agents are MCP clients. They never touch the database directly; everything funnels through a single Node process that is, itself, the lock manager. That one decision cascades: no stale reads, no write races, no distributed-lock machinery.

claim protocol

Pre-flight, not post-hoc

An agent declares intent and is granted, queued, or denied before working. Overlap is prefix-based for paths, exact for entities.

safety

Deadlock, caught early

A wait-for graph rejects any claim that would close a cycle, instantly, with a clear error, instead of two agents hanging forever.

liveness

Crash-safe leases

Every claim carries a TTL. If an agent dies mid-work, its locks auto-release. A dropped MCP session reclaims immediately.

The difference, concretely.

Every failure mode above has a direct answer. Not a best-effort convention, but an enforced guarantee.

Without · clobbering

Agent A and Agent B both write auth/config. B lands last. A’s change is gone. No error, no trace.

With · enforced writes

B can’t write without an exclusive claim A already holds. The write is rejected up front, so A’s state is never silently lost.

Without · wasted work

B spends thousands of tokens editing a module, then collides at the end and redoes all of it.

With · pre-flight claims

B asks first, learns the surface is taken, and backs off or waits, before spending a single token on the work.

Without · deadlock

A waits on B’s lock; B waits on A’s. Both hang. The run stalls until something times out, if it ever does.

With · cycle detection

The wait-for graph sees the cycle forming and rejects the closing claim immediately, with the exact cycle named.

Without · amnesia

What one run figured out (an API quirk, a decision, a config) is lost. The next fleet relearns it, paying again.

With · durable Facts

Knowledge is distilled into deduplicated, semantically searchable Facts that any future run retrieves in one call.

Without · a black box

A fleet of parallel agents is opaque. When it stalls, you have logs and guesswork.

With · a live viz

Watch who holds what, who’s blocked on whom (a real contention graph), replay the whole run by sequence, and search the memory.

The keystone: a single process holding a single database connection serializes every write, so the write-integrity guarantee is free and stale-replica races can’t exist. Most of the design falls out of that one choice.

Runs are temporary. Knowledge isn’t.

Facts live in Namespaces, not Runs. They’re embedded, deduplicated (near-identical text corroborates instead of duplicating), and retrieved by meaning. When a Run ends, a distiller (itself just an agent) reads the frozen Journal and writes what’s worth keeping.

local · default

Qwen3-Embedding

Runs in-process via ONNX. 1024-dim, private, offline after first download. No API key, no data leaving the host.

pluggable

Or bring an API

Swap to MiniMax embo-01 per Namespace. The provider is a config choice, pinned so a Namespace’s vectors always share one space.

model-agnostic

Server runs no model

Distillation is agent-side. The server owns exactly one embedding function as infrastructure, and never a chat model.

The choices that matter.

Every load-bearing decision is written down: what was traded, and why. A few worth reading:

ADR 0001

Pre-flight claims, not optimistic merge

Catch conflicts before the work, not after. Rejects the git-branch/merge model that hides shared state.
ADR 0002

One host, one process, one file

The keystone. Kills replica staleness and, as a bonus, makes MVCC unnecessary.
ADR 0003

In-memory lock manager

Turso is a durable mirror; memory is the source of truth. Blocking claims wake instantly: no polling, no pub/sub.
ADR 0005

The event loop is the mutex

Node’s single thread makes the lock manager race-free with zero explicit locking. The runtime was chosen for it.

Eight ADRs in total, plus a glossary and a full design doc. See the repository’s docs/adr/, CONTEXT.md, and DESIGN.md.
